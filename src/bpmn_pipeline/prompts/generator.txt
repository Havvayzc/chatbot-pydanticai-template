━━━ POOL STRUCTURE ━━━
- Pool-based: every distinct participant gets their own Pool (<bpmn2:participant> + <bpmn2:process>).
- External actors (customer, supplier) → own Pool.
- Internal departments sharing one process flow → Lanes within a single Pool.
- No duplicate Pools for same roles.
- All Pools fully expanded (no collapsed pools).
- Pools communicate via <bpmn2:messageFlow> inside <bpmn2:collaboration>.
- SequenceFlow stays strictly within one Pool.

━━━ NAMING ━━━
- Tasks: Object + Verb (e.g., "Antrag prüfen", "Angebot erstellen", "Teile bestellen")
- Start/End events and message events: past participle or adjective (e.g., "Anfrage eingegangen", "Auftrag abgeschlossen")

━━━ TECHNICAL REQUIREMENTS ━━━
- Every FlowNode must have <bpmn2:incoming> and <bpmn2:outgoing> child elements for SequenceFlows.
- Every Pool process needs exactly one StartEvent and at least one EndEvent.
- All SequenceFlow sourceRef/targetRef must match existing element IDs.
- If Lanes are used: every flowNode must appear in <bpmn2:flowNodeRef>.
- Syntactically correct BPMN 2.0. Escape all special XML characters (&amp; &lt; &gt;).

━━━ DI SECTION (REQUIRED) ━━━
The output MUST include a complete <bpmndi:BPMNDiagram> section.
Every process element (task, gateway, event, lane, pool) needs a <bpmndi:BPMNShape>.
Every SequenceFlow and MessageFlow needs a <bpmndi:BPMNEdge>.
Use these namespaces on <bpmn2:definitions>:
  xmlns:bpmndi="http://www.omg.org/spec/BPMN/20100524/DI"
  xmlns:dc="http://www.omg.org/spec/DD/20100524/DC"
  xmlns:di="http://www.omg.org/spec/DD/20100524/DI"

Arrange Pools horizontally (left to right) or vertically (top to bottom).
Use reasonable sizes: Pool ~1200x250px, Lane ~1200x200px, Task ~100x80px, Event ~36x36px, Gateway ~50x50px.
